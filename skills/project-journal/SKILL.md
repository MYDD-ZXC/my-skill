# Project Journal — 任务日志与项目回溯

## 你是谁

你是一个有记忆的 AI Agent。你不会忘记之前做了什么。每次任务结束时，你会写下总结；每次任务开始时，你知道最近在做什么。你的记忆通过项目目录下的规范文件持久化，跨会话保持。

## 核心文件

- **规范文件**: `project-spec.md`（项目根目录）
- **归档目录**: `archive/task-history/`（项目根目录）

---

## 规则一：任务总结（任务结束时）

### 何时触发

每次任务结束时自动执行。判断"任务结束"的信号：

1. 用户说"好的"、"可以了"、"就这样"、"完成了"、"搞定"
2. 模型完成了用户要求的所有工作
3. 用户切换到一个完全不同的话题
4. 用户明确说"先到这里"

### 记录内容

在规范文件的「任务历史」section 追加一条记录：

```markdown
| YYYY-MM-DD | {任务摘要，一句话} | 完成/进行中/中断 | {关键决策或备注} |
```

### 记录字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| 日期 | 任务执行日期 | 2026-06-08 |
| 任务摘要 | 一句话描述做了什么 | 用户管理系统竞品调研 |
| 状态 | 完成 / 进行中 / 中断 | 完成 |
| 关键决策 | 做了什么重要选择、为什么 | 确定以方案 A 为主要参考对象 |

### 同时更新「最近任务」

在规范文件的「最近任务」section 中，**插入**一条新的摘要（最新的在最上面）：

```markdown
## 最近任务
- [2026-06-08] 用户管理系统竞品分析，已完成功能对比部分
- [2026-06-05] 项目需求调研报告，初稿已交付，待用户反馈
- [2026-06-01] 项目启动，确定调研方向为用户管理系统优化
```

**保留最近 5 条**。超出 5 条时，删除最旧的那条（它已经在「任务历史」中有完整记录）。

### 关键决策的判断标准

什么算"关键决策"？记录以下内容：

1. **方向性选择** — 选择了 A 方案而非 B 方案
2. **技术选型** — 决定使用某个框架/工具/方法
3. **需求变更** — 用户调整了需求或范围
4. **排除项** — 明确决定不做什么
5. **遗留问题** — 未解决但需要后续跟进的

不需要记录的：
- 常规的代码修改
- 格式调整
- 简单的信息查询

## 规则二：任务历史管理

### 历史记录溢出处理

规范文件中的「任务历史」表格**保留最近 10 条**记录。

当超过 10 条时：

```
Step 1: 将最早的一条记录迁移到归档文件
        文件名: archive/task-history/YYYY-MM.md（按月归档）
        内容: 追加到该月的归档文件中

Step 2: 从规范文件的「任务历史」中删除该条

Step 3: 在规范文件的「任务历史」section 底部注明归档位置：
        > 更早的记录见 archive/task-history/ 目录
```

### 归档文件格式

`archive/task-history/YYYY-MM.md`：

```markdown
# 任务历史 — YYYY年MM月

| 日期 | 任务 | 状态 | 关键决策 |
|------|------|------|---------|
| 2026-06-01 | 项目启动 | 完成 | 确定调研方向 |
| 2026-06-02 | 案例搜集 | 完成 | 收集了 10 个竞品案例 |
| ... | ... | ... | ... |
```

## 规则三：目标演变追踪

### 何时触发

当 requirement-pilot 执行需求漂移检测，且用户确认调整方向时，更新规范文件的「核心目标演变」section。

### 记录格式

```markdown
## 核心目标演变
- v1 [2026-06-01]: 用户管理系统调研报告
- v2 [2026-06-08]: 用户管理系统重构方案（用户确认从调研转向执行）
```

### 记录原则

- 每次目标变更都记录，不删除历史
- 注明变更原因（用户主动调整 / 需求漂移确认 / 范围扩大）
- 版本号递增（v1, v2, v3...）

## 规则四：项目回溯

### 何时触发

当用户明确要求回溯项目历史时：
- "我们之前做了什么？"
- "这个项目的历程是什么？"
- "从头梳理一下进度"

### 回溯流程

```
Step 1: 读取规范文件中的以下 section：
        - 核心目标演变
        - 最近任务
        - 任务历史

Step 2: 如果任务历史不足 10 条，检查 archive/task-history/ 是否有更多记录

Step 3: 向用户输出项目旅程视图：
```

### 项目旅程视图格式

```markdown
## 📍 项目旅程

### 项目目标
- **初始目标** (v1, 2026-06-01): 用户管理系统调研报告
- **当前目标** (v2, 2026-06-08): 用户管理系统重构方案

### 关键里程碑
- [2026-06-01] 项目启动，确定调研方向
- [2026-06-05] 调研报告初稿完成
- [2026-06-08] 方向调整：从调研转向重构方案

### 最近进展
- [2026-06-08] 竞品分析完成，确定技术选型方向
- [2026-06-05] 调研报告初稿，待用户反馈

### 遗留问题
- 调研报告的用户反馈尚未收集
- 重构方案的数据迁移部分待补充
- 策划案的数据支撑部分待补充
```

## 自检清单

在每次对话中，对照检查：

- [ ] 任务结束时，是否写了任务总结？
- [ ] 最近任务是否保持在 5 条以内？
- [ ] 任务历史是否保持在 10 条以内？超出的是否已归档？
- [ ] 如果用户确认了需求变更，是否更新了核心目标演变？
- [ ] 用户要求回溯时，是否输出了完整的项目旅程视图？
